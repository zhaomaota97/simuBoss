# Multi-Agent 平台核心架构文档

## 核心设计理念
1. **承认模型能力的边界**：不迷信“全能大模型”，系统内所有的 Agent（包括经理）都必须是**单一职责**、**特定领域**的。
2. **人类与 AI 的绝对边界**：人类（Boss）负责创造、统筹和担责；AI（Agent）负责在既定规则和边界内执行。
3. **信息单向流动与无状态**：每一层只做自己该做的事，通过清晰的接口（如 JSON 任务树、交付物）传递信息，不感知上下文以外的世界。

---

## 整体架构图

```text
【人类世界】
👨‍💼 Boss (真实用户)
 │  ├─ 创造/销毁所有 Agent
 │  ├─ 定义职责与组建团队
 │  └─ 下发意图 / 终止任务 / 验收成果
 │
【Agent 世界】
 ├─ 🏢 团队 (Team) - 由 Boss 定义的逻辑编制模型
 │   ├─ 至少 1名 🧠 经理 (负责接收意图与逐级拆解)
 │   └─ 0~N个 🔧 工人 (负责内部具体执行)
 │
 └─ ⚙️ 调度器 (系统底层引擎)
     └─ 抓取并解析统一 DAG → 依赖检查 → 并行调度 → 注入上下文流转
```

## 每一层的详细职责

### 1. 👨‍💼 Boss（真实人类用户）
**系统唯一的“上帝”与最终责任人。**
- **创造与配置**：亲手创造每一个经理和工人，编写他们的 Prompt（定义领域和能力），并将工人分配给对应的经理组建团队。在此阶段可以为经理配置**“是否需要审批”**属性。
- **生命周期管理**：随时可以提出需求、变更需求、终止任务，随时可以调整团队配置、修改 Agent 设定甚至销毁 Agent。
- **只表达意图**：对系统只说“我要什么”，不干涉具体执行步骤。
- **最终担责**：因为所有 Agent 都是 Boss 创造和配置的，如果团队缺乏某种能力（比如需要画图但没配画图工人），这是 Boss 的失职。

### 2. 🤖 员工资源 (Employees)
系统中的原子 AI 工作单元统一称为“员工”。员工分为两种不同的职能子类型：
- 🧠 **经理 (任务拆解大脑)**：
  本身不产出具体业务交付物，专职接收业务意图，结合领域知识去输出结构化的任务树（JSON DAG）。如果遇到高度复杂的长期目标，甚至可以指派子任务给“下级经理”。
- 🔧 **工人 (无状态执行机器)**：
  单点能力的执行机器。绝对专注只做一件事（如搜库、文案、画图），无需理解整体进度。可结合自己身上的 Tool 调用外部引擎获取结果并交付。

### 3. 🏢 团队组织 (Teams)
**系统最重要的编制逻辑。团队不是一个具体的 AI，而是员工的“组合容器”。**
- **存在规则**：一个团队必定至少包含一名经理。对于复杂的超大型团队，可以允许多层级经理共存，交叉管理大量不同工种。
- **能力边界**：团队的上限取决于内部汇集的工种多样性和经理的领域架构常识。面对需求，团队的最高层经理只能调用本团队内编排进去的下属工人来进行组合；如果遇到做不了的工种，必须通过中断/抛出错误交给人类 Boss，要求人工加人。

### 4. ⚙️ 调度器（无情的状态机）
**不理解业务，只看依赖关系的剥离式引擎。**
- **事件循环**：持续扫描系统中的各种独立或团队生成的任务树，找出所有“依赖已全部满足”的原子任务。
- **调度执行**：
  - 无依赖的任务**并发启动**，精准唤醒对应的“工人”。
  - 将上游任务产出自动以 Context 或 File 引用形式流转给下游员工。
- **异常收口**：不知道员工是谁，只关心每个节点的完成态与重试上限，对一切断点予以挂起记录并请示 Boss。

## 一次完整的生命周期演示

**阶段一：Boss 创造世界（配置期）**
1. Boss 创造了【数据采集工人】、【数据分析工人】、【文案写作工人】。
2. Boss 创造了【市场调研经理】，并将上述三人分配给该经理。

**阶段二：Boss 下发意图（触发期）**
1. Boss 对【市场调研经理】说：“帮我做一份新能源汽车的竞品分析报告。”

**阶段三：经理拆解任务（规划期）**
1. 【市场调研经理】结合自己的领域知识，产出任务树（DAG）：
   - 任务 1：采集特斯拉数据（指派：采集工人，依赖：无）
   - 任务 2：采集蔚来数据（指派：采集工人，依赖：无）
   - 任务 3：对比分析（指派：分析工人，依赖：[1, 2]）
   - 任务 4：撰写报告（指派：写作工人，依赖：[3]）
2. **（审批节点）**如果【市场调研经理】配置了“需要审批”，此时系统将 DAG 呈递给 Boss。Boss 确认无误或手动微调节点逻辑后，点击“确认执行”。

**阶段四：调度器接管（执行期）**
1. 调度器看到任务 1、2 无依赖，**并行实例化**两个采集工人去干活。
2. 调度器拿到两份数据，发现任务 3 依赖满足，将两份数据打包注入给分析工人并启动。
3. 调度器拿到分析结果，注入给写作工人并启动。
4. 调度器拿到最终报告，任务树全部清空，将报告呈递给 Boss。

## 归因与迭代原则

系统出了问题，边界极其清晰，绝不互相推诿：

| 故障现象 | 责任归属 | Boss 的下一步动作 |
| :--- | :--- | :--- |
| **任务拆解逻辑混乱，步骤不合理** | **经理 Agent** | Boss 需要去修改该经理的 Prompt，补充该领域的专业知识。 |
| **任务分派给了完全不对口的工人** | **经理 Agent** | Boss 需要去强调经理的分配逻辑，或优化工人的能力描述。 |
| **任务拆对了，但工人产出质量极差** | **工人 Agent** | Boss 需要去优化该工人的 Prompt 或为其配备更好的工具。 |
| **需要画图，但经理只能派文案去画** | **Boss (人类)** | Boss 团队配置失误，需要立刻创造一个【画图工人】并加入该团队。 |

> **核心总结**：每一层都像纯函数一样工作，输入确定，输出确定。整个平台的上限，完全取决于 Boss（人类）对业务的理解深度和对 Agent 职责的拆分精度。
